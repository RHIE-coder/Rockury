# Coding Standards (Tech-Agnostic)

## Overview
프로젝트 전반에 적용되는 코드 품질/가독성/유지보수 규칙

---

## Core Principles
- **명확성**: 코드가 스스로 설명되도록 네이밍과 구조를 우선한다.
- **일관성**: 팀 합의된 스타일을 일관되게 유지한다.
- **단일 책임**: 함수/모듈은 하나의 이유로만 변경된다.
- **작은 단위**: 변경은 작고 독립적으로 유지한다.

---

## Naming Conventions (General)

### Files
- 컴포넌트/모듈은 **의미 기반 네이밍**
- 파일명 규칙은 **프로파일**에서 정의

### Symbols
- 변수/함수: **의미 있는 동사/명사 조합**
- 상수: **구분 가능한 형태** (예: SCREAMING_SNAKE)
- Boolean: **is/has/can/should** 접두

---

## Function Guidelines
- **30줄 이하** 권장, 초과 시 분리 고려
- **파라미터 3개 초과** 시 객체/구조체로 묶기
- **순수 함수 우선**: 사이드 이펙트는 경계를 분리

---

## Error Handling
- **예측 가능한 오류**는 도메인 에러로 명시
- **예상치 못한 오류**는 로깅 후 표준 응답/처리
- **에러 메시지**에 내부 구현 정보 노출 금지

---

## Imports / Dependencies
- 외부 → 내부 → 타입/스타일 순으로 정렬
- 의존성은 최소화하고, 공통 라이브러리는 중앙화

---

## Comments
- **왜(Why)**에 대한 설명만 남긴다.
- TODO/FIXME는 **이슈 또는 컨텍스트**와 함께 작성

---

## Review Checklist
- [ ] 명확한 네이밍
- [ ] 단일 책임 위반 없음
- [ ] 적절한 에러 처리
- [ ] 테스트 가능 구조
- [ ] 불필요한 의존성 없음
